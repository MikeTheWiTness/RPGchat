# PRD: RPG Chat Agent — TRPG 跑团模拟 Agent

## Problem Statement

TRPG（桌上角色扮演游戏）爱好者面临两个痛点：(1) 找不到 GM（游戏主持人）或队友时无法独自体验跑团；(2) 即使是现有的 AI 叙事工具，也无法为每个角色维护独立的上下文 — NPC 们没有自己的记忆、内心想法和独特的性格表现，导致叙事扁平、角色同质化。用户需要一个能够模拟 GM 和 NPC 的 Agent，为每个角色维护独立上下文（对话、行动、内心活动），管理环境信息的不同角色视角，并在角色与环境之间智能调度发言顺序。同时支持玩家亲自参与或全自动观剧。

## Solution

RPG Chat Agent 是一个本地运行的 TRPG 模拟引擎。用户提供战役背景后，Agent 接管 GM 角色：为每个 NPC 维护独立的角色档案和上下文，管理环境信息的多角色视角，通过判断机制在角色和环境之间智能调度发言顺序。玩家可控制自己的 PC 参与故事，或切换到全自动模式将 Agent 作为小说生成器观看。核心以统一的 JSON 动作单元作为数据高速公路，所有内容 — 玩家输入、NPC 生成、环境描述 — 都经过同一管道流转。

## User Stories

### 游戏创建与设置
1. 作为一个玩家，我想要输入战役背景文本，以便 Agent 理解世界观和初始情境开始叙事。
2. 作为一个玩家，我想要在战役背景中预定义初始 NPC 的角色档案（姓名、性格、外貌等），以便这些 NPC 从一开始就有鲜明的角色特征。
3. 作为一个玩家，我想要为我的 PC 建立角色档案，以便 Agent 在后续叙事中正确引用我的角色信息。
4. 作为一个玩家，我想要选择纯叙事模式或带骰子检定的轻量规则模式，以便根据不同 TRPG 系统的需求灵活调整体验。
5. 作为一个玩家，我想要加载预设的规则配置文件（如 COC 7th、D&D 5e），以便快速开始对应系统的游戏。
6. 作为一个玩家，我想要保存当前游戏进度到本地文件，以便中断后能随时继续。
7. 作为一个玩家，我想要加载之前保存的游戏进度，以便从中断处继续游戏。

### 玩家参与模式 — 核心循环
8. 作为一个玩家，我想要看到 Agent 在游戏开始时自动生成初始环境描述，以便立刻沉浸到故事中。
9. 作为一个玩家，我想要用自然格式输入 PC 的动作（用 `【】` 包裹对话、`（）` 包裹内心活动、纯文本表示行动），以便灵活地描述我的角色行为。
10. 作为一个玩家，我想要看到我的 PC 输入被正确解析为对话、行动和内心活动，并在剧情流中以清晰格式展示。
11. 作为一个玩家，我想要在每次角色发言后看到 Agent 判断接下来谁发言，以便了解叙事节奏的走向。
12. 作为一个玩家，我想要通过输入 `{继续}` 采纳 Agent 的发言顺序建议，以便让剧情按 GM 的判断自然推进。
13. 作为一个玩家，我想要在 Agent 推荐的发言者不是我时强行插入自己的动作单元，以便在剧情关键时刻不被打断。
14. 作为一个玩家，我想要看到 NPC 的发言、行动和内心活动（内心活动仅在角色面板中可见），以便理解每个角色的完整行为动机。
15. 作为一个玩家，我想要看到环境描述（如天气变化、场景切换、突发事件），以便感知叙事世界的动态变化。
16. 作为一个玩家，我想要在角色连续发言过多时自动插入环境描述，以便叙事不会脱离世界设定。
17. 作为一个玩家，我想要看到每个动作单元附带一个在场人员名单，以便知道哪些角色能感知到当前发生的事。

### 上下文压缩与长期记忆
18. 作为一个玩家，我想要 Agent 在到达检查点（如一天结束、场景切换）时自动压缩上下文，以便控制 LLM 的 token 消耗。
19. 作为一个玩家，我想要检查点后当前阶段的内容仍然完整展示，以便重要的近期剧情不被压缩丢失。
20. 作为一个玩家，我想要检查点时角色的档案自动更新（反映角色成长、关系变化），以便角色在长期叙事中保持一致性。

### 角色管理
21. 作为一个玩家，我想要在游戏中遇到新 NPC 时，Agent 自动为其创建角色档案，以便该 NPC 保持一致的个性表现。
22. 作为一个玩家，我想要查看任意角色的档案（性格、背景、技能）和上下文历史，以便回顾角色的发展轨迹。

### 玩家空缺模式
23. 作为一个观众，我想要切换到玩家空缺模式，让 Agent 自动推进所有角色的剧情，以便将 Agent 作为小说生成器使用。
24. 作为一个观众，我想要在自动推演时每 N 个动作单元自动暂停一次，以便我有时间阅读已生成的内容。
25. 作为一个观众，我想要在自动模式中通过 `{}` 指令干预剧情走向或调整叙事风格，以便在不参与的情况下仍能影响故事。

### 游戏机制
26. 作为一个玩家，我想要在纯叙事模式下完全由 Agent 叙述战斗和技能结果，以便享受流畅的叙事体验。
27. 作为一个玩家，我想要在轻量规则模式下通过 `{检定 技能名}` 触发技能检定，以便让骰子结果影响剧情走向。
28. 作为一个玩家，我想要看到骰子结果（如 `d100=45 对抗 侦查技能 60 → 成功`）显示在剧情流中，以便理解检定的数值过程。
29. 作为一个玩家，我想要 Agent 在 NPC 需要时自动触发技能检定，以便 NPC 的行为也有数值支撑。

### 可见性与隐私
30. 作为一个玩家，我想要环境信息中的隐藏线索仅对发现了它的角色可见，以便实现多角色视角的秘密发现机制。
31. 作为一个玩家，我想要悄悄话仅对指定角色可见，以便角色之间可以有私密交流。
32. 作为一个玩家，我想要 NPC 的内心活动不透露给其他角色，以便角色之间的信息不对称得以保持。

### 鲁棒性
33. 作为一个玩家，我想要 LLM 返回的 JSON 解析失败时自动重试最多 2 次，以便减少因格式错误导致的中断。
34. 作为一个玩家，我想要在场人员列表被 LLM 遗漏更新时通过周期性一致性校验自动修正，以便场景人员追踪长期准确。

## Implementation Decisions

### 统一动作单元 JSON 格式
所有系统内流转的内容（玩家输入解析后、LLM 生成的结果）均使用以下统一的 JSON 结构：

```json
{
  "character_id": "string | null",
  "dialogue": "string | null",
  "action": "string | null",
  "inner_thought": "string | null",
  "audience": ["character_id"] | null,
  "entered": ["character_id"] | null,
  "left": ["character_id"] | null
}
```

`dialogue`、`action`、`inner_thought` 三者至少一项不为空。环境动作单元的 `character_id` 为 `null`，且仅有 `action` 字段有意义。

### 判断机制输出 JSON 格式

```json
{
  "next_speaker": "character_id | 'environment'",
  "reason": "string",
  "force_environment": "boolean",
  "corrected_present_characters": ["character_id"] | null
}
```

### 模块划分

**深模块（核心业务逻辑，独立可测）：**
- **ActionUnitParser**: 解析 PC 输入文本生成 ActionUnit JSON；验证 LLM 返回的 JSON 格式，最多重试 2 次
- **CharacterStore**: 角色档案和角色上下文的 CRUD，支持运行时即时创建新角色档案
- **SceneTracker**: 追踪场景在场人员列表，处理 entered/left 字段更新，周期性一致性校验
- **ContextAssembler**: 为指定角色组装 LLM 生成所需的完整 prompt 上下文；为判断机制组装全知视角上下文
- **EnvironmentStore**: 管理环境信息条目，每条附带 visible_to 角色列表
- **DialogueLog**: 追加式公开对话历史，按时序存储所有对话字段
- **DiceEngine**: 骰子检定模块，支持 d100 / 3d6 / d20 表达式，匹配规则配置
- **CheckpointManager**: 触发检查点 — 压缩历史、更新角色档案、更新环境上下文

**浅模块（编排/适配层）：**
- **LLMGateway**: 统一的 LLM API 调用层
- **GameLoop**: 主循环 — 串联解析→上下文更新→判断→生成→渲染
- **PersistenceService**: 本地文件保存/加载
- **CLI / Frontend**: 用户界面

### 架构决策
- PC 在数据模型层面为列表结构，当前版本仅支持单玩家单 PC，但预留多 PC 扩展
- 本地持久化：所有游戏状态保存到本地文件系统
- 信息可见性：显式可见角色列表作为主机制（ADR-0001），后期可通过追加 visibility_condition 字段扩展
- 游戏机制：纯叙事和轻量规则集成两种模式，通过配置切换
- 发言顺序：玩家通过 `{继续}` 采纳判断机制建议，或直接输入强行插入

### 玩家输入解析规则
- `【】` 包裹 → 对话
- `（）` 包裹 → 内心活动
- 纯文本 → 行动
- `{}` 包裹 → Agent 指令（元指令，不写入角色上下文）

### 初始流程
1. 用户提供战役背景
2. Agent 生成初始环境描述（环境第一个动作单元）
3. 判断机制决定第一个行动的角色
4. 进入正常循环（PC 输入或 `{继续}` 驱动）

### 强制环境插入
连续角色动作单元数达到可配置的 N 时，判断机制必须选择环境作为下一个输出者。

### 场景在场人员维护
- 程序自动追踪每个动作单元的 entered/left 字段
- 每 K 个动作单元触发判断机制的一致性校验，以全知视角修正漂移

### 上下文压缩
检查点触发时：生成摘要替代历史 → 更新角色档案 → 更新环境上下文。当前阶段内容不参与压缩。

## Testing Decisions

### 测试原则
- 仅测试外部可观测的行为，不测试实现细节
- 深模块各自独立可测，通过其公开接口验证行为

### 需要测试的模块
1. **ActionUnitParser** — PC 输入解析的准确性（各种组合：仅有对话、仅有行动、三者都有、含 Agent 指令）、JSON 验证与重试逻辑
2. **CharacterStore** — 角色档案和上下文的创建、读取、更新、运行时即时创建
3. **SceneTracker** — entered/left 字段处理的现场列表更新、周期性一致性校验的修正逻辑
4. **ContextAssembler** — 为角色和判断机制组装的上下文字段完整性校验
5. **EnvironmentStore** — 环境条目的增删和按角色 ID 过滤可见条目
6. **DialogueLog** — 对话追加和时序读取
7. **DiceEngine** — 骰子表达式解析和检定结果计算
8. **CheckpointManager** — 上下文压缩前后的数据状态变化

### 不测试的模块（初版）
- **LLMGateway**: 依赖外部 LLM API，适合集成测试
- **GameLoop**: 编排层，依赖所有深模块的正确性，通过深模块的单元测试间接保证
- **PersistenceService**: 依赖文件系统，适合集成测试
- **CLI / Frontend**: UI 层，适合手动验证

## Out of Scope

- 多人协作（多玩家多 PC 的网络同步）
- 完整规则引擎（战斗轮、伤害计算、状态追踪）
- 动态感知条件（距离、感官类型等，已预留扩展点但不在初版实现）
- 服务端/数据库持久化（仅本地文件）
- Web UI（初版为 CLI 交互）
- 预设模组库
- 语音输入/输出
- 角色头像/立绘生成
