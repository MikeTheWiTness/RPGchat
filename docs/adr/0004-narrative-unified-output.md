# ADR-0004: Narrative 字段替代独立 dialogue/action/inner_thought 输出 + 架构演进

**日期**: 2026-06-26
**状态**: Accepted

## 背景

NPC LLM 原本输出三个独立字段（`dialogue`、`action`、`inner_thought`），由程序组装展示。这种三段式输出有以下问题：

1. **字段间割裂**：LLM 需要分别构思三个字段，导致对话和行动缺乏自然的叙事衔接
2. **动作结果轮冗余**：NPC 每次有 action 就需要额外调用一次 LLM 生成"动作结果"，增加延迟和 token 消耗
3. **叙事不连贯**：独立字段难以表达"边说边做"的自然叙事节奏

## 决策

**采用 `narrative` 字段替代独立三段式输出。** LLM 返回一段连贯的中文叙述文本，由 `split_narrative()` 在客户端解析为多个 `ActionUnit`。

### Narrative 格式规范

Narrative 是一段自由文本，使用标点符号标记不同内容类型：

| 标记 | 内容类型 | 示例 |
|------|----------|------|
| `"...""` / `"..."` / `「...」` / `『...』` | 对话 | `"该出发了"` |
| `（...）` / `(...)` | 内心活动 | `（希望别下雨）` |
| 裸文本 | 行动/动作 | `佐佐木站起身，拿起外套` |

### 解析规则

- 按标记位置切分为多个 `ActionUnit`
- 标记之间的纯文本作为 action
- 所有段共享 `character_id`、`audience`、`entered`、`left`
- `entered`/`left` 仅挂在第一段
- 若无任何标记，整段作为单一 action
- 已支持引号类型：中文双引号 `""`、英文直双引号 `"`、弯双引号 `""`、英文单引号 `'`、日式引号 `「」` `『』`

### 连带架构变更

1. **NPC 动作结果轮废弃**：NPC 的 narrative 已经包含"动作 + 感知结果"，不再需要单独的 `generate_npc_action_result()`
2. **PC 结果轮改为 `{调查}` 指令**：玩家定向指定想从 GM 获知的信息，替代自动触发的结果轮
3. **环境统一走 narrative**：但环境返回原文不解析（环境中 `（）` `""` 可能用于拟声词或注释而非标记对话/内心）
4. **环境 env_intent 增强**：`JudgmentResult` 增加 `env_intent` 字段（`atmosphere` / `break` / `transition`），让判断 LLM 明确告知环境 LLM 本轮意图
5. **剧情事件追踪三层兜底**：NPC narrative 软标记 → 判断 LLM 最终判定 → 检查点长程复核
6. **场景切换流程**：判断提议 → 玩家确认 → 检查点 → 环境生成新场景
7. **Snapshot 回滚**：每个动作单元前拍照，支持 `{回滚 N}` 回到任意历史点

## 后果

### 优势
- 叙事更自然连贯，LLM 可以控制对话和行动的节奏
- 减少 LLM 调用次数（去掉动作结果轮）
- 一次 narrative 可以包含多个对话句子和多段内心活动，粒度更灵活
- 客户端解析基于正则，零延迟
- 统一的 narrative 格式使整个系统（NPC、环境、PC 输入）对称

### 风险
- LLM 若不遵循标点格式（如用了不支持的引号类型），内容会降级为全 action
- 已扩大支持的引号类型以降低风险
- PC 端 `"..."` 格式输入因键盘限制，保留 `【】` 作为备选
