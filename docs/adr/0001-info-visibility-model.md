# ADR-0001: 信息可见性模型 — 显式角色列表 + 可选感知条件覆写

**日期**: 2026-05-27  
**状态**: Accepted

## 背景

在 RPG Chat Agent 中，环境上下文包含多条信息条目，不同的角色可能知道不同的信息（例如，某个角色发现了隐藏的线索，但其他角色不知情）。我们需要一个信息可见性模型来管理这些信息对不同角色的可见性。

## 决策

采用 **显式可见角色列表** 作为主机制，预留 **可选感知条件覆写** 作为扩展路径。

### 当前实现：显式可见角色列表

每条环境信息条目包含一个 `visible_to` 字段，直接列出能看到该信息的角色 ID 列表。角色在组装其可见环境上下文时，只获取 `visible_to` 中包含自身 ID 的条目。

```json
{
  "description": "走廊尽头有一扇微微半开的暗门",
  "visible_to": ["pc_alice", "npc_bob"]
}
```

### 后期扩展：可选感知条件覆写

当需要动态计算可见性时（如距离限制、隐身状态、感官类型等），在条目上追加 `visibility_condition` 字段：

```json
{
  "description": "走廊尽头有一扇微微半开的暗门",
  "visible_to": ["pc_alice", "npc_bob"],
  "visibility_condition": {
    "type": "distance",
    "max_range_m": 10
  }
}
```

- `visible_to` 作为静态的默认可见列表，在所有条件都失效时用作兜底。
- `visibility_condition` 为可选覆写，运行时由程序根据角色状态动态判断。
- 两种机制不冲突：`visible_to` 提供候选范围，`visibility_condition` 在此范围内进一步过滤。

## 后果

- **正面**: 数据结构简单，LLM 在 prompt 中直接使用可见角色列表，无需复杂推理。
- **正面**: JSON 字段可增量添加，现有代码无需重构即可接入感知条件。
- **正面**: `visible_to` 在无感知条件时作为兜底，防止运行时计算失败导致角色信息丢失。
- **负面**: 纯静态列表无法处理复杂感知场景（如一个角色离开房间后又回来），需要周期性一致性校验来兜底。
- **负面**: 如果感知条件大量启用，`visible_to` 字段可能与动态计算结果不一致，需要明确的优先级规则。
