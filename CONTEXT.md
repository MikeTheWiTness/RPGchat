# CONTEXT — RPG Chat Agent

## 术语表

### 角色 (Character)
分为两类：
- **玩家角色 (PC — Player Character)**: 由人类玩家输入动作单元。Agent 不为其生成内容，仅为 PC 保存上下文用于记录/追踪。
- **非玩家角色 (NPC — Non-Player Character)**: 由 Agent 控制生成动作单元。

每个角色拥有两个独立的数据结构：
- **角色档案**: 静态属性（性格、背景、外貌、技能等）
- **角色上下文**: 动态历史（动作单元记录）

### 角色档案 (Character Profile)
角色的静态属性定义，用于指导 LLM 生成该角色的发言和行动时保持角色一致性。

**基本字段**：
- `id`: 角色唯一标识
- `name`: 角色姓名
- `character_type`: 角色类型，`"pc"` 或 `"npc"`
- `personality`: 性格描述
- `background`: 背景故事
- `appearance`: 外貌描述
- `clothing`: 服饰描述
- `skills`: 技能值字典 `{技能名: 数值}`
- `attributes`: 属性值字典 `{属性名: 数值}`
- `relationships`: 与其他角色的关系描述 `{角色ID: 关系描述}`
- `notes`: 备注

**体格信息 (`physique`)**：
- `height`: 身高
- `weight`: 体重
- `build`: 体型
- `measurements`: 其他体型数据

**身份信息 (`identity`)**：
- `occupation`: 职业
- `social_status`: 社会地位
- `affiliations`: 从属组织/势力

**行为特征 (`behavior`)**：
- `habits`: 习惯
- `quirks`: 怪癖
- `mannerisms`: 言谈举止特征

**其他**：
- `intimate_features`: 私密特征（仅在角色视角可见）

**创建与更新时机**：
1. **预先定义**: PC 和初始 NPC 的角色档案在游戏开始前通过战役背景或结构化表单创建。
2. **运行时即时创建**: 新 NPC 首次出现时（经由环境或角色动作单元的 `entered` 字段引入），Agent 自动为其生成角色档案。
3. **检查点更新**: 每个检查点触发压缩后，Agent 根据当前阶段各角色的发展，更新角色档案（如性格变化、技能成长、关系变更等）。玩家在叙事中的自然语言描述（如「张三剪了短发」）会通过检查点机制间接地反映到角色档案中。检查点时，系统向玩家展示各角色的上下文摘要，由玩家决定是否清理离场角色的上下文（角色档案始终保留，仅清理动作单元历史）。
4. **玩家直接修改**: 玩家可通过 Agent 指令 `{修改角色 角色ID 字段名 值}` 直接覆写角色档案的任意字段，作为 GM 工具兜底。

### 动作单元 (Action Unit)
角色或环境的一次输出，由大模型返回的基本内容单元，包含三个可选字段：
- **对话 (Dialogue)**: 角色公开说出的台词，可选
- **行动 (Action)**: 角色做出的行为/动作，可选
- **内心活动 (Inner Thought)**: 角色的内心想法，可选

三者至少有一项不为空。PC 的动作单元由玩家输入，NPC 的动作单元由 Agent 生成。
环境也可发出动作单元（如"天色渐暗"），此时只有行动字段有意义。

### 动作结果轮 (Action Result Round) — 已废弃
~~当一个角色执行了动作（action 字段非空），紧接着需要一个专门的「动作结果轮」来描述该动作的后果~~

已被 `narrative` 叙述拆解机制替代。NPC 生成动作单元时，LLM 返回的 `narrative` 字段中已包含「动作本身 + 角色从中感知到的结果」，不再需要单独的动作结果轮。

PC 动作结果轮也已被 `{调查}` 指令替代（见下方词条）。

### 环境定位 (Environment Role)
环境是纯粹的场景管理者，职责严格限定为：
- 描写客观场景变化：时间推移、天气、光线、声音、气味、物品状态
- 引入外部事件：远处警笛、电话响起、停电等
- 初始场景的开场描述

环境**绝不**：
- 描写任何角色的行为、动作、对话或内心活动
- 替任何角色做决定
- 作为"全知 GM 旁白"叙述角色的内心或行为

### 战役背景 (Campaign Background)
故事开始前提供的战役/故事背景设定，包含世界观、初始情境、角色介绍等。由用户在创建新游戏时提供，或从预设模组加载。

核心要素：
- **世界观设定 (`world_setting`)**：世界的基本规则、历史脉络、地理环境、社会结构、力量体系等。力量体系（魔法、科技、异能、武术等）作为世界观的一部分描述，不单独拆分为字段。
- **初始情境 (`initial_situation`)**：故事开始时的具体场景和局势。
- **势力/组织 (`factions`)**：世界中的主要势力及其立场。
- **重要地点 (`important_locations`)**：关键场景地点。
- **历史事件 (`history`)**：影响当前局势的重要历史节点。

### Agent 指令 (Agent Directive)
玩家通过 `{}` 包裹的文本，向 Agent 下达的元指令（如强制剧情走向、调整叙事风格等）。不属于动作单元的内容字段，不写入任何角色上下文。仅用于影响 Agent 后续的生成行为。

### PC 输入格式 (PC Input Format)
玩家输入 PC 动作单元时使用的用户界面格式，通过标点符号区分不同内容类型：
- `【】` 或 `"..."` 包裹：**对话**（角色公开说出的台词）
- `（）` 包裹：**内心活动**（角色的内心想法）
- 无包裹的纯文本：**行动**（角色的行为动作）
- `{}` 包裹：**Agent 指令**（给 Agent 的元指令，不写入角色上下文）

示例：`我拿起长剑朝哥布林砍去"小心身后"（心里非常紧张）{接下来战斗必须获胜}`

PC 输入在进入系统后由解析器自动转换为统一的 JSON 格式，与 LLM 输出的动作单元格式保持一致。

### 动作单元 JSON 格式 (Action Unit JSON Format)
统一的内部数据表示，所有动作单元（无论来自 LLM 生成还是 PC 输入解析）均使用以下 JSON 结构：

```json
{
  "character_id": "string | null",
  "dialogue": "string | null",
  "action": "string | null",
  "inner_thought": "string | null",
  "narrative": "string | null",
  "event_flags": {"completed": ["event_id"], "clues": ["clue"]} | null,
  "audience": ["character_id"] | null,
  "entered": ["character_id"] | null,
  "left": ["character_id"] | null
}
```

字段说明：
- `character_id`: 产出该动作单元的角色 ID。环境为 `null`。
- `dialogue` / `action` / `inner_thought`: 至少一项不为空。当使用 `narrative` 字段时，这三个字段由客户端 `split_narrative()` 自动填充。
- `narrative`: LLM 返回的连贯叙述文本（替代独立的三字段）。由程序解析为多个 `ActionUnit`。详情见「Narrative 叙述拆解」词条。
- `event_flags`: NPC 对剧情事件完成/线索发现的软标记。由判断 LLM 做最终判定，不作为最终权威。
- `audience`: 受众限定，仅在需要缩小范围时填写（如悄悄话）。
- `entered`: 本动作单元中进入场景的角色 ID 列表。
- `left`: 本动作单元中离开场景的角色 ID 列表。

前端展示时，程序将 JSON 渲染为自然文本格式，对用户透明。

### Narrative 叙述拆解 (Narrative Splitting)
NPC 和环境的 LLM 通过 `narrative` 字段返回一段连贯的中文叙述文本，由 `split_narrative()` 在客户端按标点标记切分为多个 `ActionUnit`。

**标记规则**：
| 标记 | 内容类型 |
|------|----------|
| `"..."` / `""...""` / `「...」` / `『...』` / `'...'` | 对话 |
| `（...）` / `(...)` | 内心活动 |
| 标记外的裸文本 | 行动/动作 |

**解析规则**：
- 按标记出现位置依序切分，保持原文顺序
- 每段生成一个独立的 `ActionUnit`
- 标记之间的裸文本作为纯 action
- `entered` / `left` 仅挂在第一段
- 若完全没有任何标记，整段作为单一 action
- 环境返回的 narrative **不切分**——环境中 `（）` `""` 可能是拟声词或注释而非角色对话/内心标记

**PC 端兼容**：玩家输入时沿用 `【】` 标记对话（LLM 生成的 NPC/环境 narrative 使用 `""`），同时 PC 输入也支持使用 `""` 标记对话。

### JSON 解析容错 (JSON Parse Tolerance)
LLM 返回的 JSON 解析失败时，允许最多重试 2 次（共 3 次尝试），每次重试时将解析错误信息反馈给 LLM 要求修正。3 次均失败则视为生成失败，提示用户并记录错误日志。

### 回合 (Round)
所有在场角色各产生一个动作单元后，视为完成一个回合。回合是叙事节奏的**软概念**，用于感知剧情推进的大致节奏，而非严格计数的状态。

**说明**：
- 环境不是角色，环境动作单元不计入「一回合」的角色计数。
- 回合没有严格的开始/结束状态追踪，也不用于驱动游戏逻辑。
- 新角色入场、角色离场等动态变化使得精确回合边界难以定义，因此以**检查点**和**场景切换**作为真正的叙事边界。

### 角色上下文 (Character Context)
每个角色独立保存的上下文，由该角色的动作单元历史组成（对话 + 行动 + 内心活动）。

### 环境上下文 (Environment Context)
独立于角色上下文的全局环境描述，由多条环境信息条目组成。

### 环境信息条目 (Environment Info Entry)
环境上下文中的单条信息，采用通用 key-value 结构，不做强制类型约束。包含：
- `key`: 信息键名
- `value`: 信息内容
- `tags`: 标签列表（用于分类和查询）
- `visible_to`: 可见角色列表（哪些角色知道该信息）

**常见条目类型（约定而非强制）**：
- **时间**: 当前游戏内时间、日期、季节等
- **地点**: 当前场景名称、位置描述
- **天气**: 天气、温度、光线等环境条件
- **物品**: 场景中存在的可交互物品
- **氛围**: 场景氛围、情绪基调
- **状态**: 规则层面的状态标记（如「战斗中」、「潜行中」）
- **NPC**: 在场 NPC 的简要信息（作为在场人员列表的补充描述）

### 信息可见性 (Info Visibility)
采用显式标注「可见角色列表」的方式。角色只能感知标记为对自己可见的环境信息。
后期如需扩展到动态感知条件（距离、感官类型等），以可见角色列表作为默认值，感知条件作为可选覆写字段，不破坏现有数据结构。

### 公开对话历史 (Public Dialogue History)
所有动作单元中「对话」字段的时序排列。NPC 生成动作单元时可看到完整的公开对话历史，但看不到其他角色的内心活动。

**性质**：公开对话历史是**推导数据**，从各角色的动作单元历史中聚合而来，不是独立的原始数据源。实现中可作为缓存/索引以提升查询性能，但领域层面单一真相来源是各角色的动作单元历史。

### 场景在场人员 (Scene Present Characters)
维护一个全局的当前场景在场人员列表。默认情况下，每个动作单元面向所有在场人员。

**维护机制**：
1. **自动追踪**: 程序监听每个动作单元的 `entered` / `left` 字段，自动更新列表。
2. **入场**: 角色可经由自身动作单元（如"推门而入"）或环境动作单元入场。`entered` 字段中列出的角色被自动加入列表。
3. **离场**: 角色可经由自身动作单元（如"告辞离去"）或环境动作单元离场。`left` 字段中列出的角色被自动移除。

### 在场人员一致性校验 (Scene Presence Sanity Check)
每隔 K 个动作单元（K 可配置，推荐 K = 10），判断机制在输出"谁发言"的同时，对当前场景在场人员列表进行一次全量一致性校验：

- 判断机制基于其全知上下文（所有角色上下文 + 环境上下文 + 公开对话历史），重新计算当前场景应当在场的人员列表
- 若与程序追踪的列表不一致，判断机制输出 corrected_present_characters，程序以校验结果为准进行修正
- 此机制作为叙事入场/离场字段的兜底，防止 LLM 在某次输出中遗漏 `entered` / `left` 字段导致的列表漂移

### 动作单元受众 (Action Unit Audience)
每个动作单元可附带一个可选的「受众限定」字段，在全局场景在场人员列表的基础上进一步限定谁能感知到该动作单元（如悄悄话仅对特定角色可见等）。

两种机制结合：
1. 默认受众 = 当前场景在场人员列表
2. 动作单元可附带 `audience` 字段缩小范围（只减不增，不能超出场景在场人员）

**当前版本**：`audience` 作用于整个动作单元，全有或全无——只有在受众列表中的角色才能感知到该动作单元的全部内容（对话 + 行动 + 内心活动）。

**扩展预留**：未来可支持按字段细粒度控制受众（如行动所有人可见，但对话仅限受众）。届时将在动作单元上追加 `dialogue_audience`、`action_audience` 等可选覆写字段，`audience` 作为默认值保留，不破坏现有数据结构。

### NPC 生成上下文 (NPC Generation Context)
NPC 生成动作单元时可见的信息包括：
- 该 NPC 自己的角色档案
- 该 NPC 自己的角色上下文（含内心活动）
- 完整的公开对话历史
- 环境上下文中对该 NPC 可见的条目
- 当前在场人员名单

NPC **不可见**：其他角色的内心活动。

### 玩家视角 (Player Perspective)
玩家拥有**全知视角**，可以看到所有信息——无论玩家参与模式还是玩家空缺模式，玩家的视角都是全知的：
- 所有角色的动作单元（对话 + 行动 + 内心活动）
- 完整的环境上下文（包括隐藏信息）
- 判断机制的推理过程和理由

玩家的全知视角不影响 NPC 的行为 — NPC 仍然只能感知到各自可见范围内的信息。玩家只是「观察者/导演」，信息优势仅用于增强叙事理解和沉浸感。

### 判断机制 (Judgment Mechanism)
每个动作单元之后运行，判断接下来应由哪个角色/环境产生动作单元。判断结果作为**柔性建议**，玩家可接受也可打断。

判断机制可访问的上下文（类似全知 GM）：
- 所有角色的完整上下文（含内心活动）
- 完整的环境上下文
- 完整的公开对话历史

环境也可作为被判断机制选中的"发言者"。当环境产生动作单元时，只有行动字段有意义（如环境描写、时间推进、突发事件等）。

### 判断机制输出 JSON 格式 (Judgment Output JSON Format)
判断机制（也是 LLM 调用）的输出结构：

```json
{
  "next_speaker": "character_id | 'environment'",
  "reason": "string",
  "force_environment": "boolean",
  "env_intent": "atmosphere | break | transition | null",
  "proposed_scene_change": "string | null",
  "completed_events": ["event_id"] | null,
  "discovered_clues": ["clue"] | null,
  "corrected_present_characters": ["character_id"] | null
}
```

字段说明：
- `next_speaker`: 推荐的下一个动作单元产出者（角色 ID 或 `"environment"`）。
- `reason`: 简短理由，用于日志和调试。
- `force_environment`: 若已通过泊松随机决定强制环境，此字段为 `true`，此时 `next_speaker` 必须为 `"environment"`。判断 LLM 被告知此信号后自决 `env_intent`。
- `env_intent`: 当 `next_speaker` 为 `"environment"` 时指定环境介入意图：
  - `"atmosphere"`: 烘托气氛——仅感官细节，不引入新事件/人物
  - `"break"`: 打破僵局——引入外部事件推动剧情
  - `"transition"`: 场景过渡——暗示场景即将切换
  - 非环境轮时为 `null`
- `proposed_scene_change`: 判断提议的场景切换目标描述。为 `null` 时不提议切换。
- `completed_events`: 判定为已完成的关键事件 ID 列表。为 `null` 时无新完成事件。
- `discovered_clues`: 判定为已发现的线索列表。为 `null` 时无新线索。
- `corrected_present_characters`: 仅在周期性一致性校验触发时填写（每 K 个动作单元），提供修正后的完整在场人员列表。非校验轮次为 `null`。

### 强制环境插入 (Forced Environment Insertion) — 已废弃
~~为避免角色连续动作单元过多导致叙事脱离环境控制，设定一个最大连续角色动作单元数 N（可配置）。~~

已被泊松递增概率 + `env_intent` 机制替代。判断 LLM 在 prompt 中被告知当前连续角色轮数，自行决定环境介入时机和意图类型（atmosphere / break / transition）。不设硬性最大连续轮数。

### 初始流程 (Initial Flow)
故事开始时：
1. 用户提供战役背景
2. Agent 基于战役背景生成初始环境描述（环境第一个动作单元）
3. 判断机制运行，决定第一个行动的角色
4. 进入正常循环

### 发言顺序 (Speaking Order)
发言顺序是**柔性的建议**而非强制规则。判断机制根据情境推荐下一个发言者，玩家永远可以打断或改变顺序。

玩家拥有最高发言权：
- 若玩家输入 `{继续}`，则采纳判断机制的推荐，由被推荐的 NPC/环境产生**一个**动作单元（下一步）。
- 若玩家输入 `{到我}`，则连续产生 NPC/环境动作单元，直到轮到 PC 发言时停下（继续到下一轮 PC 回合）。
- 若玩家直接输入 PC 动作单元，则玩家的输入优先于判断机制，允许玩家强行插入。

**界面映射**：CLI 中通过 `{继续}` 和 `{到我}` 指令触发；未来 GUI 版本可对应为两个按钮（"继续下一步"、"继续到我"）。

### 玩家确认原则 (Player Confirmation Principle)
玩家对重大叙事决策拥有最高决定权。LLM 可主动提议，但以下操作必须经玩家确认才能执行：

| 操作类型 | 是否需要确认 | 说明 |
|---------|-------------|------|
| 场景切换 | ✅ 需要确认 | 改变叙事场景属于重大变化，LLM 可提议，玩家确认后生效 |
| 检查点触发 | ✅ 需要确认 | 检查点触发上下文压缩和角色档案更新，LLM 可提议，玩家确认后生效 |
| 时间快进 | ✅ 需要确认 | 跳跃式时间推进（如「三天后」）改变叙事节奏，玩家确认后生效 |
| 角色档案重大变更 | ✅ 需要确认 | 性格大转变、技能跃迁等重大档案变化需玩家确认 |
| 角色入场/离场 | ❌ 自动处理 | 场景内的角色流动通过 `entered`/`left` 字段自动处理，无需确认 |
| 骰子检定 | ❌ 玩家主动触发 | 检定由玩家主动发起，不需要额外确认 |

玩家也可通过 `{修改角色}`、`{切换场景}` 等 Agent 指令直接强制执行任何操作，绕过确认流程。

### 上下文压缩与检查点 (Context Compression & Checkpoint)
为控制上下文字数，每隔一定进程对历史内容进行压缩提炼：
- **检查点 (Checkpoint)**: 触发压缩的叙事节点，如"一天结束"、"一个场景切换"。不同的 TRPG 模组可自定义检查点的粒度（按天、按场景、按章节等）。
  - **触发方式**：Agent 可主动提议检查点，玩家也可通过 `{检查点}` 指令手动触发。所有检查点需经玩家确认后生效（见玩家确认原则）。
  - **场景切换与检查点的关系**：场景切换必然触发检查点，但检查点不一定是场景切换（如一天结束也是检查点，但场景未变）。
- **摘要 (Summary)**: 对检查点之前的内容进行提炼压缩，生成结构化摘要替代原始上下文的细粒度历史。原始内容在压缩后可丢弃或归档。
- 压缩不影响当前未完成阶段（当前一天/当前场景）内的完整上下文展示。
- **检查点附带操作**: 每个检查点除了压缩历史外，还触发以下操作：
  - 角色档案更新（各角色档案根据当前阶段发展进行更新）
  - 环境上下文更新（清理过期条目，写入新的全局环境状态）
  - 玩家确认：系统展示各角色上下文摘要，由玩家决定是否清理离场角色的上下文（角色档案始终保留）

### 玩家空缺模式 (Player-Absent Mode)
核心运行模式之一，适用于剧情自动推演、小说生成或世界观模拟场景。玩家以「导演/GM 视角」参与，拥有最高决定权，但不控制具体角色。

**核心特征**：
- 不创建 PC，所有角色均为 NPC，由 Agent 控制生成动作单元
- 剧情自动推演，无需玩家逐次触发
- **检查点自动暂停**：推演到检查点时自动暂停，等待玩家确认后继续（给玩家干预和消化剧情的机会）
- **玩家确认原则仍然适用**：场景切换、检查点、时间快进、角色档案重大变更等操作需玩家确认
- 玩家可通过 `{}` Agent 指令随时干预剧情走向、节奏和风格
- **骰子检定系统不启用**：只有运势系统调节 NPC 行为倾向
- **运势系统正常运作**：每个检查点刷新各角色运势，影响 NPC 行为

**与玩家参与模式的对比**：

| 维度 | 玩家参与模式 | 玩家空缺模式 |
|------|-------------|-------------|
| PC | 有，玩家控制 | 无 |
| 触发方式 | 玩家 `{继续}` / `{到我}` | 自动推演，检查点暂停 |
| 骰子检定 | 启用（玩家侧） | 不启用 |
| 运势系统 | 启用（NPC 侧） | 启用（所有角色） |
| 玩家角色 | 参与者（控制 PC） | 导演/观察者 |
| 玩家确认 | 适用 | 适用 |

### 运行模式 (Operating Mode)
Agent 支持两种运行模式，在创建游戏时选定：
1. **玩家参与模式 (Player-Present Mode)**: 存在 PC，玩家控制 PC 输入，判断机制通过 `{继续}` 触发。默认模式。
2. **玩家空缺模式 (Player-Absent Mode)**: 无 PC，全自动推演，玩家仅通过 Agent 指令干预。

### 玩家模型 (Player Model)
- **当前版本**: 单玩家 + 单 PC。一个玩家控制一个角色，其余角色均为 NPC。
- **多玩家/多 PC**：短期内不考虑。当前所有设计均围绕单玩家单 PC 优化，不做多角色兼容。未来如确有需要，可作为重大版本迭代重新评估。

### 本地持久化 (Local Persistence)
游戏会话数据保存到本地文件系统，包括：
- 战役背景
- 所有角色的角色档案和角色上下文
- 环境上下文
- 公开对话历史
- 检查点和摘要
- 场景在场人员列表当前状态
- 规则配置（如启用）

支持保存/加载游戏进度，中断后可继续。

### 角色预设 (Character Preset)
预先定义好的角色档案模板，保存在 `presets/characters/` 目录下。**预设是跨战役可复用的**，不属于任何特定模组——例如一个「现代医生」角色预设可以在所有现代背景的战役中使用。

- **导入**：通过 `{导入角色 名称或路径}` 指令加载预设到当前游戏
- **列出**：通过 `{列出角色预设}` 查看可用的角色预设

### 调查指令 ({调查} Directive)
当 PC 执行了一个动作，玩家需要获知该动作的客观结果（如"查看信件内容"、"搜索房间"等），但玩家不知道客观事实，通过 `{调查 描述}` 指令定向请求 GM 描述结果。

- 替代旧的自动 PC 动作结果轮
- 玩家明确指定想了解的信息，LLM 聚焦回答
- 结果由环境/场景叙述者输出，融入叙事流
- 仅在有实际需求时消耗一次 LLM 调用，而非每次有 action 都触发

**示例**：
```
玩家: 我拆开信封
系统: (无自动结果轮)
玩家: {调查 信中内容}
系统: [环境] 信纸上只有一行字："明晚八点，老地方见。" 字迹潦草，似乎写得很匆忙。
```

### 世界观预设 (World Preset)
预先定义好的战役背景模板，保存在 `presets/worlds/` 目录下。**预设是跨战役可复用的**——例如「经典西幻世界观」预设可以作为多个不同冒险模组的基础。

- **导入**：通过 `{导入世界观 名称或路径}` 指令加载预设到当前游戏
- **列出**：通过 `{列出世界观预设}` 查看可用的世界观预设

### 模组 (Module)
完整的可玩冒险包，由世界观、初始情境、初始 NPC、预设角色等组件组合而成。模组是「开箱即用」的游戏体验，玩家加载后可以直接开始游戏。

模组与预设的关系：
- **预设**是细粒度的可复用组件（单个角色、单个世界观）
- **模组**是粗粒度的完整游戏包（世界观 + 初始情境 + 初始 NPC + 推荐角色）
- 一个模组可以包含多个预设，一个预设可以被多个模组复用

### AI 生成预设（待开发）
通过 LLM 根据简短描述自动生成完整的预设内容，保存到预设库供以后复用。支持三种粒度：

1. **角色预设**：输入简短描述（如「酒馆老板，中年，和蔼可亲」），生成完整的角色档案
2. **世界观预设**：输入简短描述（如「经典西幻世界观」），生成完整的战役背景
3. **完整模组**：输入简短描述（如「经典西幻，半精灵男主角的冒险故事」），一次性生成世界观 + PC 角色卡 + 初始 NPC + 初始情境，可直接开始游戏

### 骰子引擎 (DiceEngine)
`dice.py` 中的纯逻辑模块，提供 d20 投骰和技能检定功能。核心方法：
- `roll_d20(modifier)`: 投 1d20 + 加值，返回 `DiceRollResult`（含表达式、每次投骰值、加值、总值）
- `check_skill(skill_value, dc)`: 执行一次技能检定，返回 `CheckResult`（含成功/失败、暴击/大失败标记、检定描述）

自然 20 = 暴击（必定成功），自然 1 = 大失败（必定失败）。

### 快照与回滚 (Snapshot & Rollback)
每个动作单元处理前，系统自动保存完整游戏状态的快照（Snapshot），包含：
- 所有角色上下文
- 环境条目
- 对话日志
- 场景在场状态
- 最后判断结果
- 导演意图栈
- 剧情追踪器状态
- 角色档案字典
- 战役背景

玩家可通过指令管理快照：
- `{历史}` 或 `{快照列表}`：查看最近 20 条快照
- `{回滚 N}`：恢复到序号 N 的快照状态，撤销该序号及之后的全部动作单元

最多保留 200 条快照。回滚后所有受影响的状态（包括动态创建的角色档案和在场人员列表）一同恢复。

### 剧情事件追踪 (Plot Event Tracking)
三层兜底机制，确保关键剧情事件的完成不会被遗漏：

1. **NPC 软标记 (event_flags)**：NPC 生成 narrative 时可选输出 `event_flags: {completed: [...], clues: [...]}`，作为"我认为我完成了某事件"的第一人称信号。非权威，可误报。

2. **判断 LLM 最终判定**：判断机制在生成 `JudgmentResult` 时，基于全知上下文评估当前叙事状态，输出 `completed_events: [...]` 和 `discovered_clues: [...]`。这是主要的权威判定来源。

3. **检查点长程复核**：检查点时由检查点 LLM 对整个阶段做回顾，确认/修正判断的标注，补标遗漏的事件和线索。作为最终兜底。
`rules.py` 中的模块，处理 PC 的 `{检定 技能 [DC]}` 指令。解析指令文本、查找 PC 技能值、调用 `DiceEngine` 执行检定、生成可读的检定结果描述。纯叙事模式下拒绝检定请求。

### 自动模式控制器 (AutoModeController)
`auto_mode.py` 中的模块，在玩家空缺模式下驱动剧情自动推演。循环调用判断机制 → 生成 NPC/环境动作单元，在遇到待确认事项（检查点提议）时自动暂停，等待玩家确认后继续。

### 确认管理器 (ConfirmationManager)
`confirmation.py` 中的状态机模块，管理玩家确认流程。支持四类确认请求（场景切换、检查点、时间快进、角色档案重大变更），提供 `propose()` → `confirm()` / `reject()` 的标准状态流转。

### 运势字段 (CharacterContext.fortune)
`CharacterContext` 上的 `fortune` 字段（默认 `"normal"`），存储角色当前运势等级。由检查点触发刷新（通过 `FortuneSystem.roll_fortune()`），NPC 生成时通过 prompt 注入运势行为倾向描述。详见"运势系统"词条。

### 预设加载器 (PresetLoader)
`preset_loader.py` 中的模块，负责从 `presets/` 目录加载角色预设、世界观预设和模组。核心函数：
- `list_characters()` / `load_character_preset(name)`: 列出/加载角色预设
- `list_worlds()` / `load_world_preset(name)`: 列出/加载世界观预设
- `list_modules()` / `load_module(name)`: 列出/加载完整模组
- `build_pc_from_preset(name, name_override, personality_override)`: 将角色预设转换为 PC 档案
- `list_module_characters(module_name)`: 列出模组内角色

### 语言说明
本项目的 UI 界面、LLM prompt 模板、预设内容和文档的主要语言为**简体中文**。预设文件名也使用中文（如 `前刑警佐佐木健吾.json`）。代码标识符和注释使用英文。
### 游戏机制集成 (Game Mechanics Integration)
支持两种运行层级，可在创建游戏时选定并在运行时切换：

1. **纯叙事 (Pure Narrative)**: 不涉及任何骰子或数值系统。Agent 根据角色档案和上下文自由叙述剧情走向。战斗、交涉等场景的效果完全由 Agent 判断。

2. **轻量规则集成 (Lightweight Rules)**: 集成 d20 骰子/技能检定模块。
   - **骰子系统**: 统一使用 d20 系统（roll-over，即投骰结果 + 技能加值 >= 难度等级 = 成功）。
   - **技能值含义**: `CharacterProfile.skills` 中存储的是**技能加值**（modifier），与 d20 结果相加后与 DC 比较。
   - **难度等级 (DC — Difficulty Class)**: 检定的目标数值，由情境难度决定。
   - **触发方式**: 玩家通过 Agent 指令 `{检定 技能名 [DC]}` 触发检定；NPC 的动作单元可由 Agent 自行决定触发检定。系统执行骰子后将结果数值插入叙事流。
   - **角色档案扩展**: 启用规则集成时，角色档案额外包含属性值和技能加值。

后期可扩展为**完整规则引擎 (Full Rules Engine)**，纳入战斗轮、伤害计算、状态追踪等完整规则逻辑。

### 运势系统 (Fortune System)
NPC 行为的随机性调节机制，通过程序随机数 + 提示词引导的方式实现，不依赖 LLM 原生的 temperature 参数。每个 NPC 在每个检查点周期内有一个固定的运势等级，影响其行为倾向。

**运势等级**（五级制，d100 投点分档）：
- **大吉** (95-100, 5%)：超常发挥、灵感爆发、意外顺利
- **吉** (75-94, 20%)：状态不错、偏积极、小成功
- **平** (26-74, 49%)：正常状态
- **凶** (6-25, 20%)：状态不佳、小失误、偏消极
- **大凶** (1-5, 5%)：严重失误、完全搞错、出糗

**运作方式**：
1. 每次检查点刷新时，为每个在场 NPC 独立投一次 d100，确定其下一阶段的运势等级
2. 生成 NPC 动作单元前，根据其运势等级在提示词中注入对应的行为倾向描述
3. 运势只影响 NPC 的行为倾向，不直接决定成败（叙事层面的调节，而非数值层面的硬规则）

**与骰子检定的关系**：
- 骰子检定（d20 + 技能加值 vs DC）是**玩家侧**的严谨机制，明确判定成功/失败
- 运势系统是**NPC 侧**的软调节，增加叙事多样性和意外感
- 两者独立运作，互不影响
